# Agent Reference - Design Document

## Concept

"Introduced by My Agents" — AI 에이전트가 사용자와의 실제 협업 경험을 기반으로 객관적인 레퍼런스 체크 리포트를 작성하는 스킬.

### 왜 만드는가

- 이력서의 자기소개는 과장되기 쉽고, 단점을 장점처럼 포장하는 게 관례
- 바이브 코딩 시대에 나와 가장 대화를 많이 하고, 내 작업 스타일을 가장 잘 아는 존재가 AI 에이전트
- 동료에게 레퍼런스 체크를 하듯, 에이전트에게 하는 것
- 에이전트는 실제 작업 과정을 함께했기 때문에, 구체적 근거 기반의 객관적 분석이 가능

### 포지셔닝

- 상용 서비스 아님, 오픈소스
- skills.sh 포맷 준수 → CLI로 다양한 AI 에이전트에 설치 가능
- 영문 기반 (README는 영문 + 한국어 제공)
- 리포트 출력 언어는 사용자 세션 언어를 따름

## Phase 3: 포트폴리오 사이트 (별도 프로젝트)

종합 리포트를 기반으로 Astro 포트폴리오 사이트를 생성한다.
agent-reference 스킬과는 분리된 별도 프로젝트로 진행.

- Hero: 한 줄 소개 + 에이전트들의 한마디 요약
- Agent Reviews: 제품 리뷰처럼 각 에이전트 페르소나가 남긴 후기 카드
- Projects: 프로젝트별 기여도 + 에이전트가 본 특징
- Work Style: 분석 차원별 시각화
- Blog: 설계 고민 요약 (세션 대화에서 자동 추출)
- FAQ: 면접관이 궁금할 만한 질문들 미리 답변
- Raw Data: 원본 리포트 링크 (투명성 확보)

## Design Decisions Log

이 설계 과정에서 내린 주요 결정들.

| 결정 | 이유 |
|---|---|
| 스크립트가 아닌 스킬 문서(프롬프트) 방식 | 각 에이전트가 자기 세션 컨텍스트에서 자연스럽게 분석. 별도 파싱 코드 불필요 |
| 에이전트 페르소나 고정하지 않음 | 사용자마다 사용 에이전트가 다르고, 같은 에이전트도 맥락에 따라 역할이 달라짐 |
| 리포트 = 사용자 프로필 + 프로젝트별 분리 | 이력서 구조와 동일. 사람 소개 / 프로젝트 경험을 자연스럽게 분리 |
| 템플릿은 러프하게 | 에이전트마다 고유한 관찰이 포함되어야 함. LLM이 문서 파악을 잘하므로 엄격할 필요 없음 |
| 포트폴리오 사이트 별도 프로젝트 분리 | 스킬은 분석/리포트에 집중. 사이트는 리포트를 소비하는 별도 프로젝트 |
| 영문 기반, 리포트 출력은 세션 언어 | skills.sh 글로벌 생태계 대응. 실사용은 각자 언어로 |
| 신뢰도 메타데이터 포함 | 세션 3개 vs 50개의 분석 깊이 차이를 명시 |
| Review History 섹션 추가 | 사용자 피드백을 리포트 하단에 기록하여 투명성 확보. 인라인 면책 조항 대신 구조화된 이력 |
| 도구 이름이 아닌 사용자 행동 패턴 기술 | Read/Bash 등 에이전트 도구명이 아니라, 사용자의 지시 패턴과 작업 스타일로 서술 |
