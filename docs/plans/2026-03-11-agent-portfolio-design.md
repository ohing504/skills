# Agent Portfolio — Design Document

Phase 3 of the agent-reference project. A skill that generates a personalized portfolio site from agent-reference reports.

## Concept

Take the reports generated by the `agent-reference` skill (Phase 1 & 2), combine them with the user's resume and GitHub data, and generate a personalized Astro portfolio site deployed to GitHub Pages.

The AI customizes the site's concept, color palette, typography, and emphasis points based on report analysis — so each developer's portfolio reflects their actual working style as observed by their AI collaborators.

## Skill Overview

```
Skill: agent-portfolio
Location: skills/agent-portfolio/

Input:
├── agent-reference reports (required)     — Phase 1 & 2 outputs
├── Resume / self-introduction (optional)  — user-provided
├── GitHub profile data via gh CLI (optional)
└── Existing portfolio materials (optional)

Output:
├── Astro portfolio site → GitHub Pages
├── GitHub Profile README.md (optional)
└── Deployed and live at {username}.github.io
```

## Tech Stack

- **Framework**: Astro (content-driven static site, Markdown/MDX native)
- **Skill dependency**: `astrolicious/agent-skills@astro` (1.9K installs, official Astro team)
- **Styling**: TBD (Tailwind CSS likely, fits Astro ecosystem)
- **Charts**: TBD (Work Style radar chart, GitHub activity visualization)
- **Deployment**: GitHub Actions → GitHub Pages

## Customization Approach (Level 2)

AI analyzes the agent-reference reports and proposes a site concept:

1. **Analyze reports** — identify dominant traits (e.g., "methodical architect", "rapid prototyper", "full-stack generalist")
2. **Propose concept** — suggest a design direction that reflects the person's style
3. **Apply theme** — color palette, typography, section ordering, emphasis points
4. **User approval** — present the concept before generating the site

The layout uses reusable components that AI recombines per concept. Not a fixed template, but not freeform generation either — structured variety.

### Example Concepts

| Report signals | Proposed concept |
|---|---|
| Methodical, architecture-first, documentation-heavy | Clean, structured, whitespace-rich. Blueprint aesthetic |
| Fast iteration, many projects, diverse tech stack | Dynamic, colorful, project-gallery focused |
| Deep domain expertise, few large projects | Minimal, content-heavy, case-study style |

## Site Sections

| Section | Required | Description |
|---------|----------|-------------|
| **Hero** | Yes | One-line intro + agent summary quotes |
| **About** | Optional | From resume/self-introduction if provided |
| **Agent Reviews** | Yes | Persona cards — product-review style (CTO, PM, teammate) |
| **Projects** | Yes | Per-project cards with agent-observed contributions |
| **Work Style** | Yes | Analysis dimensions visualized (radar chart or similar) |
| **GitHub Activity** | Optional | Contribution calendar, language distribution, stats |
| **FAQ** | Optional | Interviewer Q&A from aggregated reports |
| **Blog Topics** | Optional | Agent-extracted blog topic seeds |
| **Raw Data** | Optional | Full report texts for transparency |

## Deployment Architecture

### Option B: Private repo (Recommended)

```
{username}/portfolio (private repo)
├── src/                          # Astro source
├── public/                       # Static assets
├── reports/                      # agent-reference report files (private)
│   ├── 2026-03-claude-code/
│   │   ├── user-profile.md
│   │   ├── project-financial.md
│   │   └── project-agentfiles.md
│   └── 2026-09-cursor/
│       ├── user-profile.md
│       └── project-newproject.md
├── resume.md                     # User's resume (optional)
├── .github/workflows/deploy.yml  # Build & deploy to Pages
├── astro.config.mjs
└── package.json
```

- Reports stay private in the repo
- GitHub Actions builds Astro → deploys to GitHub Pages
- Only user-selected reports appear on Raw Data page
- Site is public at `{username}.github.io`

### Option A: Public repo (Simple)

```
{username}.github.io (public repo)
├── (same structure as above)
```

- Simplest setup — repo IS the site
- All reports visible in git history
- Fine if user has no private repo content in reports

### Getting Started Flow (for README)

```
1. Run agent-reference skill → generate reports
2. Run agent-portfolio skill → scaffolds Astro site + copies reports
3. Review AI-proposed concept → approve or adjust
4. Site is built and pushed to GitHub
5. Enable GitHub Pages in repo settings
6. Live at {username}.github.io
```

## Update Workflow

When the user has new reports (new machine, new agent, new projects):

1. Run `agent-reference` again in the new environment → new report files
2. Copy new reports to `reports/` directory in portfolio repo
3. Run `agent-portfolio` again → AI merges old + new data, regenerates site
4. Push → GitHub Actions rebuilds and deploys

Reports accumulate over time. The skill reads all reports in `reports/` and synthesizes across time periods.
